# Perspecta Workflow — Skill-Based Discovery & Delivery

**Date:** 2026-06-04
**Status:** Approved (design), pending implementation plan
**Repo:** `~/Documents/GitHub/perspecta-workflow`
**Depends on:** Phase 1.5 (marker, auto-color, Set node type) — merged to `main`.
**Relates to / supersedes parts of:** Phase 1.6 (discovery & invocation,
`2026-06-04-workflow-discovery-and-invocation-design.md`) — this design replaces
that phase's slash-command delivery surface with skills, while keeping its
registry (`INDEX.md`), its `start`-node `trigger:`/`purpose` authoring
convention, and its name-collision handling.

## Purpose

Make Perspecta workflows **discoverable and automatically invokable by agents**
working in the vault — and runnable by users — without anyone having to remember
a command. An agent should proactively know (a) what workflows exist, (b) what
each does and when to trigger it, and (c) how to run one.

The plugin becomes a **delivery vehicle** for the agent-facing files that make
this work: it installs them when missing and updates them when a newer version
ships (the generic capability) or when a canvas changes (per-workflow content).

The tool/script and alternative-data-source layers a workflow might use are
explicitly **out of scope** here; this design only establishes detection and
automatic use. The chosen architecture (a skill folder per workflow) is the
natural future home for those layers.

## Two-tier skill model

Both tiers live in the vault's `.claude/skills/`, where Claude Code
auto-discovers project skills.

| | **Tier 1: Generic skill** | **Tier 2: Per-workflow skills** |
|---|---|---|
| Path | `.claude/skills/perspecta-workflow/SKILL.md` | `.claude/skills/<workflow-name>/SKILL.md` |
| Owner | Plugin (bundled = source of truth) | Generated from canvases |
| Content | Reusable know-how: how to discover, walk, and execute *any* Perspecta workflow | Identity + trigger + canvas path + pointer to the generic skill |
| Updated when | Plugin version increases | Any canvas changes (regenerated) |
| Source of truth | The plugin bundle | The canvas + `start`-node frontmatter |
| Update mechanism | Version-stamped reconcile on activate (§ Tier 1) | Full regenerate on activate + manual command (§ Tier 2) |

The generic skill is **thick** (holds the walking know-how once); per-workflow
skills are **thin** (identity + path + "see `perspecta-workflow` for how to
walk"). This keeps generated text minimal, so drift is minimal.

The two tiers map onto the two delivery problems:

- The **generic skill** is the *plugin-as-delivery-vehicle* problem:
  install-if-missing, update-if-version-newer.
- The **per-workflow skills** are the *generate-from-canvas* problem:
  regenerate so they never drift from the workflows they advertise.

## Tier 1 — generic skill delivery

The install-if-missing / update-if-newer behavior.

### Bundling

The canonical `perspecta-workflow/SKILL.md` ships **inside the plugin bundle**.
esbuild inlines it as a string constant — the plugin folder under
`.obsidian/plugins/` is the only thing that travels to a vault, and a loose
sidecar file the plugin would have to locate at runtime is fragile, so the skill
text is embedded in `main.js`. Its frontmatter carries
`perspecta_version: <plugin manifest version>`, stamped at build time from
`manifest.json`.

### Reconciliation on activate (`onload()`)

```
read .claude/skills/perspecta-workflow/SKILL.md
├─ missing                        → write bundled  (install)
├─ installed_version <  bundled   → overwrite      (update)
├─ installed_version == bundled   → leave alone    (no-op, no git noise)
└─ installed_version >  bundled   → leave alone    (never downgrade)
```

- Version compare is **semver** on the `perspecta_version` frontmatter field.
- A write happens **only** on install or genuine upgrade — never on an unchanged
  launch — so Obsidian start-up does not churn git.
- A version-parse failure on the *installed* stamp is treated as "older" and
  overwritten (self-heals a corrupted stamp).

### Contract

The generic skill is **plugin-owned**. Hand-edits are not preserved across a
version bump. The skill's own header documents this:

> _Generated by Perspecta Workflow vX — do not hand-edit; changes are
> overwritten on plugin update._

## Tier 2 — per-workflow skill generation

Each marked canvas → one generated skill at
`.claude/skills/<workflow-name>/SKILL.md`.

### Naming

`<workflow-name>` derives from the canvas filename. The Phase-1.6
name-collision handling carries over: if two canvases share a filename, the
generator disambiguates by appending a short path segment, and the embedded
canvas path remains the unambiguous reference.

### Content (single source of truth = the canvas)

- `name` ← canvas filename
- `description` (the auto-trigger text) ← `start`-node frontmatter `trigger:`,
  falling back to `purpose` (first body line of the start note), then to the
  name
- canvas path ← the file location, embedded so the agent knows what to walk

### Generated skill body (thin)

```markdown
---
name: <workflow-name>
description: <trigger from start node — what makes the agent auto-reach for it>
perspecta_generated: true
perspecta_source: <canvas path>
---
Use the `perspecta-workflow` skill for how to walk a workflow. This one:
1. Walk the canvas at `<canvas path>` via MCP `workflow_start(...)`.
2. If the perspecta-workflow MCP tools aren't connected this session,
   read the canvas + node-notes and walk them manually.
```

The execution contract is **MCP-first with a documented file-fallback**: the
agent uses the real engine (`workflow_start` / `workflow_current` /
`workflow_advance`) when the server is connected, and degrades to reading the
canvas + node-notes directly otherwise. The detailed how-to-walk know-how lives
once in the generic skill; per-workflow skills point back to it.

### Generated-marker discipline

- Every generated per-workflow skill carries `perspecta_generated: true`.
- On regeneration, the plugin only ever **overwrites files carrying that
  marker** — a hand-authored skill in `.claude/skills/` is never touched.
- **Orphan pruning:** on a full regenerate, any `perspecta_generated: true` skill
  whose source canvas no longer exists (deleted, or marker removed) is removed.

### When it runs

- **On activate** — full scan of all marked canvases → regenerate every
  per-workflow skill + prune orphans. This catches canvases edited **outside**
  Obsidian: an agent editing a canvas via MCP, or a self-improving workflow that
  rewrites itself (see the self-improving-workflows design), both of which would
  otherwise leave a stale skill description.
- **On the manual command** — "Perspecta: Rebuild workflow skills" runs the same
  scan on demand.

## Registry coexistence & ambient awareness

The Phase-1.6 `_agents/workflows/INDEX.md` registry and `CLAUDE.md` pointer
remain; they serve a different need than the skills and share the same authoring
convention.

**Division of labor:**

- **Per-workflow skills** = *proactive triggering*. The agent matches a user
  request to a workflow via the skill's `description`, through normal skill
  auto-discovery. This is the "automatically used" path.
- **`INDEX.md` registry** = *enumeration*. A human- and agent-readable list of
  all workflows (name, purpose, when-to-use, node count) for "what workflows
  exist here?" and for a no-argument listing.

Both are generated from the **same** `start`-node `trigger:`/`purpose`
convention in the **same scan**, so one authoring act feeds skill descriptions
*and* the registry, and they cannot drift from each other.

**`CLAUDE.md` pointer** — kept but demoted. With per-workflow skills
auto-triggering, the pointer is no longer the primary discovery mechanism; it
becomes a one-line backstop:

> This vault defines Perspecta workflows as skills and in
> `_agents/workflows/INDEX.md`.

The plugin installs/updates this pointer **idempotently** as a marked block
(replace-in-place), creating `CLAUDE.md` if absent.

**Slash commands** (`/perspecta:workflow`, `/ppa:workflow`) from Phase 1.6 are
**out of scope** for this design. Skills cover both proactive
(description-triggered) and explicit ("use the person-brief workflow")
invocation, making the command layer redundant for now. Deferred rather than
built as a third delivery surface.

## Architecture

### Core (`@perspecta/core`) — pure, fs-agnostic, unit-tested

- `registry.ts` (from Phase 1.6) —
  `summarizeWorkflow(canvas, startNote) → WorkflowSummary { name, purpose,
  trigger, canvasPath, nodeCount }`.
- `skillgen.ts` (**NEW**) — pure functions:
  - `renderWorkflowSkill(summary) → string` — the per-workflow `SKILL.md`.
  - `renderRegistry(summaries) → string` — the `INDEX.md`.

  No fs, no `node:` imports. Takes parsed inputs, returns strings. This is the
  heavily-unit-tested heart shared by the MCP server and the Obsidian plugin.

### Plugin (`obsidian-plugin`) — the fs / lifecycle seam

- `src/skills/genericSkill.ts` — the esbuild-inlined generic-skill string +
  `reconcileGenericSkill()` (the version-compare install/upgrade logic).
- `src/skills/syncWorkflowSkills.ts` — scan marked canvases → call core
  `renderWorkflowSkill` → write `.claude/skills/<name>/`, prune orphaned
  `perspecta_generated` skills, update `INDEX.md`, update the `CLAUDE.md`
  pointer block.
- `src/main.ts` wiring — call `reconcileGenericSkill()` and
  `syncWorkflowSkills()` on `onload()`; register the "Rebuild workflow skills"
  command.
- All writes reuse the existing `ObsidianFileSystem` / `app.vault.adapter`.

### Error handling

- All reconciliation is **best-effort and wrapped**: a malformed canvas or a
  single bad `start`-note is skipped with a `Notice`, never aborts the whole
  sync and never blocks plugin load.
- Missing `start` node or no `trigger:` → fall back (purpose → name); never
  fail.
- `.claude/skills/` missing → create it.
- Version-parse failure on an installed generic skill → treat as "older",
  overwrite (self-heals a corrupted stamp).
- Writes happen **only** on a real change (install / upgrade / content-diff), so
  a clean launch produces no git churn.

## Testing

- **core `skillgen` + `registry`** — fully unit-tested with mocked canvas + note
  strings → expected `SKILL.md` / `INDEX.md`. Fallback chains covered: no
  `trigger:`, no start note, name collision.
- **plugin `reconcileGenericSkill`** — version-matrix tested
  (missing / older / equal / newer / corrupt-stamp) against a mocked vault.
- **plugin `syncWorkflowSkills`** — generate + prune-orphan + don't-touch-a
  hand-authored (unmarked) skill.
- **manual** — agent-session verification that a workflow's `trigger:` actually
  causes proactive skill discovery, and that the MCP-first / file-fallback paths
  both walk a workflow.

## Scope

### In (this design)

- Tier 1: bundled, version-stamped generic `perspecta-workflow` skill +
  reconcile-on-activate (install / upgrade / never-downgrade).
- Tier 2: per-workflow `SKILL.md` generation from canvases, full regenerate on
  activate + "Rebuild workflow skills" command, `perspecta_generated` marker +
  orphan pruning.
- core `skillgen.ts` (`renderWorkflowSkill`, `renderRegistry`).
- `INDEX.md` registry generation (shared scan) + idempotent `CLAUDE.md` pointer.
- Reuse of the Phase-1.6 `start`-node `trigger:` / `purpose` authoring
  convention.

### Out (deferred)

- Workflow-usable **tools / scripts** bundled in a skill folder (the future
  layer the per-workflow skill folder is designed to host).
- Alternative **data sources** beyond md-files / vault-memory.
- Slash-command delivery surface (`/perspecta:workflow`) — superseded by skills.
- Save-driven (per-canvas) incremental regeneration via the existing watcher —
  activate-scan + manual command is the chosen trigger; a watcher could come
  later.
- Autonomous in-plugin LLM execution (Phase 3 — unchanged).

## Open questions / deferred

- **Name uniqueness:** inherited from Phase 1.6 — workflow `name` derives from
  the canvas filename; collisions across folders are disambiguated by appending
  a short path segment, with the embedded canvas path as the unambiguous
  reference. Acceptable for the current vault (no clash).
- **MCP connection assumption:** per-workflow skills prefer the
  `workflow_*` MCP tools but document the canvas + node-note file-walk fallback
  for sessions where the server isn't connected.
